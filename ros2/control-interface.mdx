The generic ROS 2 adapter converts Sentinel control output into standard ROS 2 messages. Configure the topics and mappings around the controller you already use.

<Warning>
  **Start the joint-state publisher before the Sentinel stack.** Sustain at least 100 Hz and include every controlled joint. If no fresh state arrives for 500 ms, Sentinel triggers an E-stop.
</Warning>

## Required measured state

Publish `sensor_msgs/JointState` continuously at 100 Hz or faster. Higher rates are acceptable. Consistency matters more than choosing a specific rate above that minimum. State must continue while the robot is idle or disarmed.

Populate:

- `header.stamp` with the measurement time
- `name` with stable joint names
- `position` in radians
- `velocity` in radians per second when available
- `effort` in newton-metres when available

Sentinel matches state by joint name. Message order may change between samples, but names and values must remain aligned within each message.

```bash
ros2 topic hz /robot/joint_states
ros2 topic echo /robot/joint_states --once
```

The current safety timeout is 500 ms. A gap longer than that faults the adapter and escalates to an E-stop. Do not increase or disable the timeout to work around an unreliable publisher or DDS connection.

## Robot description

Sentinel uses the URDF for joint names, kinematics, and limits. Provide it from one of these sources:

| Source | Use when |
| --- | --- |
| URDF or xacro file | The model ships with the integration |
| `robot_description` parameter | An existing `robot_state_publisher` owns the description |
| Transient-local `robot_description` topic | The description is already distributed over ROS 2 |

Make the description available before the control pipeline starts. Give each independently controlled system its own namespace and description source.

## Arm commands

Choose the output your controller accepts.

| Output | Message | Joint identity |
| --- | --- | --- |
| Named trajectory | `trajectory_msgs/JointTrajectory` | `joint_names` in the message |
| Forward position | `std_msgs/Float64MultiArray` | Fixed configured array order |

<Tip>
  For `ros2_control`, use `forward_command_controller/ForwardCommandController` with the position interface. Do not place a `joint_trajectory_controller` between Sentinel's continuous targets and the hardware.
</Tip>

All joint positions use radians. Joint velocities use radians per second.

For `JointTrajectory`, process the current target point according to the controller contract. For `Float64MultiArray`, declare the exact slot order in the integration configuration.

<Warning>
  Never infer a forward-position array order from the current `JointState` message. A `JointState` publisher may reorder names. The command array order is fixed by the controller configuration.
</Warning>

## Joint mapping

The adapter can translate between Sentinel joint names and external controller names. It can also apply an affine transform:

```yaml
command:
  topic: /external/arm/joint_trajectory
  qos: reliable
  joint_remap:
    shoulder_pan_joint: joint_1
  joint_transform:
    wrist_3_joint:
      scale: -1.0
      offset: 0.0

state:
  topic: /external/arm/joint_states
  qos: sensor
  joint_remap:
    joint_1: shoulder_pan_joint
  joint_transform:
    wrist_3_joint:
      scale: -1.0
      offset: 0.0
  stale_timeout_s: 0.5
```

`external = scale × sentinel + offset` for commanded position. State uses the inverse mapping. Keep the public contract in SI units; do not use transforms to introduce degrees.

## QoS

Set QoS independently for command and state topics.

| Preset | Typical use |
| --- | --- |
| `reliable` | Controller command subscriptions |
| `sensor` or `best_effort` | High-rate state publishers using sensor-data QoS |
| `default` | Publishers and subscribers using ROS 2 defaults |

QoS must be compatible at both ends. If a topic appears in `ros2 topic list` but data never arrives, compare reliability and durability before changing the adapter code.

## Other command interfaces

### Gripper

A gripper can use `JointTrajectory`, `Float64`, or `Float64MultiArray`. Map Sentinel's normalized opening command to the device range. Publish measured gripper state when the hardware provides it.

### Mobile base

Subscribe to `geometry_msgs/Twist` for mobile-base commands. Linear velocity uses metres per second. Angular velocity uses radians per second. Stop safely when the command stream stops.

### Neck, hand, elevator, and PTZ

The generic adapter also supports configured neck, dexterous-hand, elevator, and PTZ outputs. Start with the arm and measured-state path. Add these capabilities only after the base integration is stable.

## Validate before Sentinel

<Steps>
  <Step title="Inspect the description">
    Confirm that the expected joint names and limits appear in the URDF.
  </Step>
  <Step title="Verify continuous measured state">
    Confirm that every controlled joint is present and that state continues while the controller is idle.
  </Step>
  <Step title="Send a small controller-native command">
    Test the command topic without Sentinel in a cleared workspace.
  </Step>
  <Step title="Stop the command publisher">
    Confirm that the controller holds or stops according to the robot's validated watchdog behavior.
  </Step>
  <Step title="Match QoS and domain settings">
    Run the checks from the same container and domain that will run Sentinel.
  </Step>
</Steps>
